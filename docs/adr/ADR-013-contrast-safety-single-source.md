# ADR-013: 对比度保障策略——设计 token 单一事实源 + 双向机械门控

- 状态：Accepted（人类 2026-07-10 要求"建立样式策略，保证不再出现同类问题"）
- 适用层：项目实现（BoardX 专属：模板只带模式引用）
- 日期：2026-07-10
- 作者：coord-architecture
- 关联：uiux-standards.md（对比度架构节）、ADR-012（机械门控优先于人肉纪律的同款哲学）

## 背景：同一个"黑底黑字"bug 在修复后 24 小时内复发

| 日期 | 现场 | 直接原因 |
|---|---|---|
| 2026-07-09 | Rooms/Boards Create 按钮黑底黑字 | `cn()` 合并时 tailwind-merge 不认识自定义字号 `text-13`，把它当**文字颜色**处理，与 `text-primary-foreground` 判为同组 → 后者被吞 |
| 2026-07-09 修复 | 把字号清单**手抄**进 `extendTailwindMerge` 的 font-size 组 | 清单来源 = 当时 tailwind.config.ts 的档位快照 |
| 2026-07-10 | AVA 研究类型按钮（`variant="default"` + `text-12`）黑底黑字 | 手抄清单里**没有 12**——代码里实际用了 `text-12`（64 处）但它既不在 tailwind fontSize 表（类不生效）也不在 merge 清单（继续吞配色） |

全仓盘点进一步发现：`text-12/14/18/20/21` 共 83 处使用**全部在表外**——字号表与真实用法早已系统性漂移，07-09 的修复只是把其中一份手抄副本对齐到了另一份手抄副本的旧快照。

## 根因（架构层，不是哪一行代码）

`text-*` 前缀在 Tailwind 中同时承载**字号**与**文字颜色**两种语义，歧义消解完全依赖
tailwind-merge 的 font-size 类组登记。于是同一份字号表实际存在于**三个地方**：

1. `tailwind.config.ts` fontSize（类是否生效）
2. `lib/utils.ts` 的 merge 登记（配色类是否被吞）
3. 代码里的实际用法（作者以为哪些档位存在）

三者靠人肉保持一致——**手抄清单必然漂移，漂移的后果恰好是最隐蔽的可访问性事故**
（黑底黑字在开发者自己的屏幕上往往是黑底白字，因为他没触发 merge 路径）。

## 决策

### D1. 字号档位单一事实源：`lib/font-scale.ts`

一个文件导出 `FONT_SIZE_SCALE`；`tailwind.config.ts` 与 `lib/utils.ts` 都从它 import，
**手写第二份清单被制度性消灭**。新增档位改一处，类生效与 merge 识别自动同步。
apps/web 与 apps/devportal 同策略（各自持有该文件；跨 app 共享待设计系统包化时合并）。

### D2. 双向机械门控（对齐 ADR-012"修工具不教育使用者"）

- **正向（运行时守卫）**：`lib/utils.test.ts` 遍历 `FONT_SIZE_KEYS`，逐档断言
  `cn("text-primary-foreground text-<N>")` 两类共存——任何进了单源的档位自动被
  测试覆盖，"登记了类但 merge 不认识"这个缝隙从此测不过。
- **反向（lint 门控）**：`lint-design.sh §1.6` 扫描代码里全部 `text-<数字>` 用法，
  发现表外档位直接红（含文件行号）——"用了不存在的字号"从静默失效变成 CI 失败。
  两个方向合起来：**表内的必然正确，表外的必然被拦**。

### D3. 组件层规则（写入 uiux-standards）

配色**永远成对**且**来自 variant**：需要 `bg-primary` 的地方用
`<Button variant="default">` / `<Badge>` 等封装（它们内部已成对写好 bg+fg），
不在页面层手拼 `bg-primary` + 手补 foreground。页面层 `className` 只允许追加
布局/间距/字号，不允许出现落单的语义色 bg-*。

### D4. 历史用法的处置：入表而非改码

83 处表外字号（12/14/18/20/21）**全部登记进单源**而不是改写成邻近档位——
它们是作者的真实意图（此前类不生效、字号一直在回落继承），入表让意图生效。
副作用：这些位置的实际渲染字号会从"继承值"变为"声明值"，属于**修复而非回归**。

## 后果

- 同类事故的两个成因（清单漂移、表外用法）都被机械门控覆盖，人肉记忆退出。
- 新增字号的成本从"改三处且容易漏"降为"改一处，漏了测试/lint 会叫"。
- 代价：`text-12` 等 83 处的渲染字号变化（意图恢复），UI 走查时按 D4 理解。

## 经验教训

1. **修 bug 时问一句"这个清单还有没有别的副本"**——07-09 的修复对齐了两份副本
   中的一份，第三份（代码用法）没人看，24 小时后从那里复发。
2. **`text-*` 的字号/颜色双语义是本设计系统最尖的暗桩**，任何触碰字号表的改动
   必须同时想到 merge 组——现在由单源结构代想。
3. 与 ADR-012 同款教训的样式版：**能被机器判定的一致性，绝不交给人肉纪律。**
