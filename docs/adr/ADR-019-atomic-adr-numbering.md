# ADR-019: atomic-adr-numbering（ADR 编号原子取号，new-adr 命令）

- 状态: Accepted
- 适用层：方法论（可移植）
- 日期: 2026-07-19
- 作者：coord-architecture
- 关联：#660（phase-id 撞号收口的同款根因与修法）、ADR-018（本 ADR 编号本身撞过号的当事者）

## 背景

`docs/adr/README.md` 顶部一直有一句"新 ADR 编号从 ADR-NNN 起"的静态提示，写 ADR 的人
读这句话手翻表格数出下一个号、手写文件。这句提示**只存在于已合入 main 的文本里**，
在飞分支彼此不可见——两个人在各自分支上同一时刻都读到"新 ADR 从 ADR-018 起"，都
认领了 ADR-018（#778 spec-ref-closed-loop 与 #730 harness-v2），直到合并才发现撞号。

这与 #660 的 phase-id 撞号是**完全同构的根因**：分配权威只活在事后才可见的已合并
文本里，在飞分支之间没有任何机制能让后来者看见"号已经被占了"。#660 用
`phases/roadmap.yaml`（机器可读、占号即登记）解决了 phase id；ADR 编号这次撞车
证明同一类问题会在任何"手翻文档数下一个号"的地方复现，只是换了个壳。

## 决策

新建 `pnpm harness new-adr --title "<标题>" [--id ADR-NNN] [--layer methodology|project]`：

- **默认自动取号**：扫 `docs/adr/README.md` 索引表现有 id 的数字部分，取 max+1，
  3 位零填充。
- **显式 `--id` 必须无冲突**：撞索引表已有条目**或** `docs/adr/` 下已有同 id 前缀
  的文件（双来源检测，覆盖"文件建了但忘记登记索引表"的历史情况）即报错退出，并
  提示下一个可用号——不再允许静默占号。
- **占号即登记**：scaffold 文件的同一次运行里，把新增条目**原子写回**
  `README.md` 索引表（表格插入一行；`--layer` 给定时，同时把 id 追加进"## 适用层"
  分类清单对应的一行，跟随既有的紧凑写法——首项带 `ADR-` 前缀，后续只写数字）。
  后到的在飞分支在 rebase/merge `README.md` 时会看见真实的文本冲突，而不是各自
  静默选中同一个号后到合并才发现。
- 取号/冲突检测抽为纯函数 `.harness/scripts/lib/adr-id.ts`（不碰文件系统，13 个
  单测覆盖：解析、取号、双来源冲突检测、前缀不误伤）。

### 顺带修的一个模板 bug

`adr.template.md` 的标题行原是 `# ADR {{ADR_ID}}: {{ADR_TITLE}}`，硬编码了一个
"ADR "前缀；但 `ADR_ID` 变量本身已经带着"ADR-"前缀（如"ADR-019"），过去全靠人手
复制模板时顺手改掉这处重复，从未有自动化真正跑过这条路径。这次让 `new-adr` 真的
调用 `renderTemplateFile` 后，第一次跑就生成出"# ADR ADR-019: ..."——自动化把
一个存在多年、但每次都被人手动纠正所以从未暴露的 bug 现出原形。改成
`# {{ADR_ID}}: {{ADR_TITLE}}`，与既有文件（如 ADR-018 的 `# ADR-018: ...`）
的真实格式对齐。

## 后果

- 新 ADR 从此有唯一的起号入口，README.md 的"新 ADR 从 X 起"静态提示同步删除
  （改为指向本命令），过期提示本身就是这次事故的直接成因，不能只修事故不修病灶。
- 历史 ADR（0001~ADR-018）不受影响，编号/内容均未改动。
- 代价：多了一层脚本依赖——纯手写 ADR 文件仍然可行（模板本体没变），但不再是
  推荐路径；`--layer` 未指定时索引表主题栏留占位符，写正文时人工补全，机器不
  替人拍板层级归属。

## 我们什么情况下会改主意

若未来 ADR 数量/团队规模大到需要更完整的元数据（作者、review 状态机、废弃
自动检测）——纯文本 README 表格会不够用，那时候值得评估迁到结构化格式（如
YAML front matter 索引 + 生成 README），当前规模（<20 个 ADR）不构成升级理由。
