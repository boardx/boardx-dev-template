# UX 细节对齐 UI 原型

## 背景 / 为什么做
对照 `phases/requirements/ava/uc-ava-001~010` 与 `phases/requirements/BoardX UI Prototype V1.html`
里的 AVA 相关锚点，发现几处原型明确要求、但在当前实现里容易被简化/忽略的交互细节。这些是体验层
面的打磨项，依赖阶段 01-04 的功能先落地（比如语音可视化依赖语音输入本身先做出来）。

## 原始需求
- 作为用户，我录音时看到的应该是随音量变化的动态波形/时长展示（原型用字符波形 `▁▃▅▇▅▃▁` +
  计时表达），而不是一句静态的"录音中…"文案。
- 作为用户，我发起 Deep Research 后，应该先看到针对我主题的澄清问题，确认/调整后看到研究计划
  并确认，然后才进入执行——这是两个独立的、我可以交互确认的步骤，不能从输入主题直接跳到报告。
- 作为用户，当前聊天已经有消息时，如果 Agent 切换受限，我应该在界面上直接看到"不可切换"的禁用
  提示，而不是点了之后才告诉我不行。
- 作为用户，打开一个无效/已关闭/为空的分享只读链接时，应该看到对应场景的清晰提示（Loading /
  Invalid chat session / 链接已失效 / No messages），而不是只有"一切正常"这一种展示状态被验证到。

## 验收线索
- 语音录音 UI 有可视的音量强度反馈（哪怕是简化的柱状/波形表达，不要求音频频谱级精度）和实时计时。
- Deep Research 的澄清问题面板与计划确认面板是两个可点击确认/调整的独立步骤，用户必须显式确认
  计划才进入执行阶段。
- Agent 切换入口在"线程已有消息且切换受限"场景下呈现禁用视觉状态（而不是可点击后才报错）。
- `/chatShare/:threadId` 只读页的四种状态（Loading / Invalid / Unavailable-已关闭 / Empty-空线程）
  各自有对应的 e2e 覆盖，不能只测"正常展示消息"这一种路径。

## 范围与边界
- 本阶段要做：以上四项体验细节。
- 明确不做：不做原型里非 AVA 区域的其它体验打磨（如 Board 分享面板、Survey 报告面板等，那些不
  属于本阶段范围）。

## 已知约束 / 依赖
- 依赖 `03-voice-input-stt.md`（语音输入本身要先存在，才能打磨可视化）与
  `02-deep-research-realization.md`（澄清/计划确认依赖后端真实生成，不能在硬编码 stub 上做假的
  两步确认）。
- F04/F05 的分享只读页四态覆盖可以独立于其它 feature 先行推进（现有实现已支持四种状态，本项主要
  是补齐 e2e 覆盖，风险较低，可优先安排）。

## 切分提示
- 建议顺序：先做分享只读页四态覆盖（风险最低、依赖最少），其余三项跟随各自依赖的功能一起交付，
  不必单独等到最后集中做。
