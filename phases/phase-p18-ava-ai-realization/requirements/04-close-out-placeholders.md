# 接通已知占位

## 背景 / 为什么做
以下几处已经在 UI 上呈现给用户、e2e 也断言其"保持某个状态"，但背后没有真实逻辑——这些不是新范围，
是已声明 `passing` 的 feature 里藏着的占位实现，优先级高于任何新范围扩展：
- F04（分享聊天）："发送到我的邮箱"按钮永久 disabled，无 onClick。
- F07（AI 设置）：Agent 选择器数据源是硬编码的 2 个 Agent（`AVA_AGENT_OPTIONS`），真实数据依赖
  `phase-p11-ai-store` 的 F03（订阅/使用→带入 AVA），该 feature 目前仍 `in_progress`。
- F08（附件）：消息流里的附件渲染只是"图标 + 文件名"chip，没有图片缩略图、音频播放器、文件预览。
- F11（消息结果操作）："发送到 Board""发送邮件"按钮无 onClick，e2e 只断言它们保持 disabled。

## 原始需求
- 作为用户，点击分享面板里的"发送到我的邮箱"，应该真的收到一封包含分享链接的邮件，并看到成功/
  失败提示，而不是按钮点了没反应。
- 作为用户，当我团队的 AI Store 里有真实订阅的 Agent 时，AVA 的 Agent 选择器应该列出这些真实
  Agent，而不是固定的 2 个占位项。
- 作为用户，AI 回复里包含图片附件时应该看到缩略图，包含音频附件时应该看到可播放的音频卡片，包含
  文件附件时应该有预览条，而不是全部退化成同一种文件名 chip。
- 作为用户，点击 AI 消息下的"发送到 Board"，应该能把该消息内容真的发送到我有编辑权限的白板上并
  给出成功提示；点击"发送邮件"应该真的发出邮件。

## 验收线索
- F04：点击"发送到我的邮箱"后，能观察到邮件发送请求真实发出（真实发信或至少接入邮件服务的
  API 调用，走既有的邮件服务能力，不是新建一套邮件基础设施），失败时有独立于"复制链接"的错误提示。
- F07：`phase-p11-ai-store` 的 F03 转 `passing` 后，Agent 选择器改为从真实订阅数据源查询；如果
  本阶段执行时 F03 仍未 passing，先完成"数据源改造为可接入真实数据的接口形态"，并把"切换到真实
  数据源"标注为依赖 p11-F03 的后续小项，不能虚报为已完全对齐。
- F08：至少覆盖图片（缩略图+点击预览）、音频（可播放卡片）两种富渲染；文件预览条视工作量可放在
  本阶段或标注为下一迭代，但图片和音频是硬指标。
- F11：目标 Board 有编辑权限时"发送到 Board"成功写入内容并有成功提示；无权限时给出权限不足提示。
  邮件发送的成功/失败提示与 F04 的邮件发送共享同一套底层能力（避免重复造轮子）。

## 范围与边界
- 本阶段要做：以上四项占位全部接通为真实行为（F07 视 p11-F03 进度可能是"接口就绪，待切换"）。
- 明确不做：不新增分享/发送邮件之外的通知渠道（如短信）；不做 F08 里除图片/音频外的所有附件类型
  富预览（如 Office 文档在线预览），文件类附件保留基础的下载/预览条即可。

## 已知约束 / 依赖
- 依赖的能力平面：CAP-DATA（F04 分享）、CAP-AI（F07 Agent 数据）、CAP-FILE（F08 富渲染）。
- F07 的真实数据切换依赖外部阶段 `phase-p11-ai-store` 的 F03，不受本阶段控制，需要在 sprint 规划
  时把这条依赖显式排进去（可能需要等 p11 一个 sprint 后再排本阶段这个子项）。

## 切分提示
- 建议拆成 4 个独立 feature（F04 邮件分享、F07 Agent 真实数据、F08 附件富渲染、F11 发送到
  Board/邮件），彼此依赖很弱，可以并行推进。
