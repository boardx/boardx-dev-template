# AI 层去 stub 化

## 背景 / 为什么做
`packages/ai/src/gateway.ts` 目前只注册了一个 `stubProvider`（匹配 `stub:` 前缀的 modelId），
`packages/ai/src/graph.ts` 的「LangGraph 风格编排」实际是一个单函数 `runChatGraph()` 调用。
`packages/ai/package.json` 零运行时依赖，全仓库搜索 claude/anthropic/gemini/gpt/openai/litellm
只在注释里出现过。这意味着 AVA 聊天壳（F01）、编辑重生成（F03）、AI 设置里选的模型（F07）
即使用户体验上"能聊"，本质都是模板化确定性回显，不是真实 AI。这是所有下游功能可信度的地基问题。

## 原始需求
- 作为用户，我发给 AVA 的消息应该收到针对我输入内容的真实 AI 回复，而不是模板回显。
- 作为工程团队，我们需要至少接入一个真实模型 provider（参考老代码支持 Claude/Gemini/GPT 多模型，
  本阶段至少做通一个，不要求对齐老代码全部三家）。
- 作为工程团队，`packages/ai/` 里"LangGraph 编排""LiteLLM 网关"这类命名如果本阶段仍不做实，
  必须去掉误导性描述，不能让注释暗示一个不存在的架构。
- 停止生成（Stop）目前依赖 `AbortController`，真实模型接入后需要确认这个机制仍然有效（流式请求
  可以被真实中断，而不是等 stub 回显完）。

## 验收线索
- 在非 mock 配置下，同一句话问两次真实模型，回复内容应体现出模型的实际推理而非固定模板（不要求
  内容完全一致，但要能看出是真实生成）。
- 切换 F07 里的模型选择器，选中不同真实模型后发送消息，请求确实路由到了不同 provider（可通过
  日志/请求参数验证），而不是像现在一样全部指向同一个 stub。
- 真实 provider 不可用（网络错误/超时/限流）时，用户能看到明确的失败态提示，且已输入的内容不丢失
  （沿用 F01 现有的"生成失败展示失败态且不丢用户输入"行为）。
- 现有的 stub 路径仍然保留（用于 e2e 测试等不依赖真实供应商额度的场景），新增真实 provider 是新增
  能力而非替换掉 stub。

## 范围与边界
- 本阶段要做：至少一个真实 provider 接入 + 可切换 + 失败态处理 + 停止生成在真实流式场景下验证。
- 明确不做：不要求做到老代码那样的多模型（Claude/Gemini/GPT）全覆盖，也不做模型层面的成本/配额
  管理（如有配额限制，走现有的计费错误处理，不在本阶段新增）。

## 已知约束 / 依赖
- 依赖的能力平面：CAP-AI。
- 需要真实模型的 API key/凭证配置（走 `.env`，不进代码库，具体供应商由实现时决定并在 evidence 里
  记录使用的是哪个 provider）。
- e2e 验证需要区分「mock 模式」（CI 用，快速稳定）与「真实模式」（至少一条烟雾级验证），避免真实
  API 调用拖慢/拖垮日常验证流水线。

## 切分提示
- 可以拆成 2 个 feature：(1) 真实 provider 接入 + 网关路由改造，(2) 失败态/停止生成在真实模型下的
  行为验证。优先级：(1) 优先，(2) 依赖 (1)。
