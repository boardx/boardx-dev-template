# 原始需求 — Billing F04 拉齐（Phase p21）

## 背景 / 为什么做
p14-F04（升级/管理个人计划）标 `passing`，其中"订阅升级弹窗"和"credits 模式切换"两项核实属实，
但**"额度不足触发升级弹窗"这条因果链在代码里根本不存在**：`apps/web/app/(app)/ava/page.tsx`
里研究请求收到 402（额度不足）时只会 `setSendError` 展示错误文字，从来不会调用
`setBillingOpen(true)`；页面上那个看起来相关的"AI credits"提示横幅（`ai-low-credits-prompt`）
是**无条件常驻渲染**的静态 UI，点击它的按钮才会打开弹窗，但这个点击和"是否真的额度不足"完全
无关。现有 e2e 测试制造了"已覆盖"的假象（一个测静态横幅点击、一个测错误文字），但没有一个测试
真正验证"服务端返回 402 → 弹窗被打开"这条完整链路——因为这条链路本来就不存在。

## 原始需求（用户故事 / 大白话都行）
- 作为用户，当我因为 credit 余额不足被服务端拒绝执行某个 AI 操作（如 Deep Research）时，我想要
  系统能引导我去升级/购买额度，而不是只看到一段看不懂下一步该干嘛的错误文字。

## 验收线索（可观察的成功是什么样）
> 产品需要先拍板走哪条路径（见下方切分提示），两条路径的验收线索不同：
- 路径 A（补真实因果链）：触发一次真实的 402 额度不足响应后，升级弹窗自动打开（或至少那条
  "AI credits"提示横幅从"无条件展示"变成"仅在额度不足时展示/高亮"），有 e2e 能真正模拟 402
  场景并断言弹窗/横幅的联动反应。
- 路径 B（如实降级描述）：`feature_list.json` 的 F04 `user_visible_behavior`/notes 明确改写为
  "额度提示横幅为静态常驻 UI，不随实际额度状态变化；用户可随时手动点击进入升级流程"，不再声称
  "触发"这个因果关系；对应 e2e 测试名和断言同步改名，不再叫"额度不足触发"。

## 范围与边界
- 本阶段要做：F04 额度不足触发的因果链修正（走路径 A 或 B，见切分提示）；同时补充 notes 明确
  声明"仅支持单一货币（USD）/不支持退款/不支持自动续费管理"这几项相对 oldcode 收窄的范围（现状
  notes 对此没有说明，容易被后来者误以为遗漏）。
- 明确不做：credit 扣减防透支 `WHERE balance + delta >= 0` 的 guard——AI 消费扣减钩子目前还没
  接入任何实际触发路径（p9/p12 的事），本阶段只在 notes 里记录这个风险点，不提前实现用不到的
  防护；发票/退款/多币种——oldcode 本身也没做完，不算新增遗漏，本阶段不补。

## 已知约束 / 依赖
- 依赖的能力平面：CAP-PAYMENT。
- billing 是 registry.yaml 里 rev-security 的 required_for 之一，即便本 feature 看起来只是
  UI 联动修正，只要涉及升级弹窗触发条件也建议过一次 rev-security（避免引入新的绕过路径，比如
  客户端伪造"额度不足"状态骗过后续可能新增的联动逻辑）。

## 切分提示（给 requirement-author 的建议）
- 这是本阶段里唯一一个**需要产品先拍板方向**（路径 A vs B）才能定验证命令的 feature，
  requirement-author 生成时如果拿不准，应该在 feature 的 notes 里显式标注"待产品确认路径 A/B"，
  不要自行替产品做决定；默认可以先按路径 B（如实改描述，工作量小、不引入新行为）起草，
  产品后续如果要路径 A 再单独加一个 feature。
- 优先级低于 auth/team 的安全类 feature，可以放在同一 wave 或稍后一个 wave，不阻塞前两者。
