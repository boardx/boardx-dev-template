# BoardX Survey 原始需求与现状基线

版本：1.0  
日期：2026-07-14  
文档类型：现状基线 + 目标产品需求 + 验收标准  
适用对象：产品经理、设计师、前后端工程师、AI Agent、测试与验收人员

## 1. 文档目的

本文档将 BoardX Survey 当前代码反推为一份可执行的产品需求基线，使人工团队和 AI Agent 能够一致回答以下问题：

1. 系统服务谁，解决什么问题。
2. 当前已经实现哪些能力，哪些只有界面或模拟实现，哪些尚未实现。
3. 每项功能应如何工作，与哪些数据、接口和上下游模块发生关系。
4. 什么结果才算完成，如何通过页面、接口、数据库和导出文件进行验收。
5. 后续迭代应按什么优先级补齐生产闭环。

本文档不是对某一张截图的说明，也不把原型展示、Mock 数据或按钮存在视为功能完成。

## 2. 事实来源与状态定义

### 2.1 事实来源优先级

发生冲突时按以下顺序判断系统真实能力：

1. 数据库模型、数据访问层和 API 行为。
2. 可运行的页面交互。
3. 自动化测试和可重复验证命令。
4. 历史设计文档。
5. 截图、演示文案和静态占位内容。

### 2.2 状态定义

| 状态 | 定义 |
| --- | --- |
| 已实现 | 前端、接口、数据持久化和主要异常处理形成闭环，并存在可重复验证方式。 |
| 部分实现 | 已有页面或部分接口，但存在 Mock、临时数据、缺少持久化、缺少权限、缺少正式导出或缺少关键异常处理。 |
| 待实现 | 当前代码没有形成可用能力，或现有行为与目标要求明显不一致。 |

## 3. 产品定位

BoardX Survey 是一套 AI 原生问卷与专业报告系统。它帮助业务人员从研究目标或参考材料出发，完成问卷设计、报告规划、发布回收、答卷分析和专业报告交付。

产品的核心不是“生成一份表单”，而是建立以下闭环：

```text
明确研究目标
  -> 生成或选择问卷
  -> 人工确认题目与分类
  -> 设计报告规划
  -> 发布并回收真实答卷
  -> 按分类分析数据
  -> 生成有证据的专业报告
  -> 导出并支持管理决策
```

### 3.1 核心价值

- 降低专业问卷的设计门槛。
- 在收集数据前明确报告结构，避免“先收数据、后想怎么分析”。
- 用 AI 提升题目设计、分类、数据解读和报告撰写效率。
- 保证报告结论来自真实答卷，并显式区分事实、推断、假设和限制。
- 让网页预览、PDF、Word 和图片导出保持同一份报告内容。

### 3.2 非目标

- 本阶段不建设完整 BI 平台或通用数据仓库。
- 不支持 AI 未经用户确认自动发布问卷。
- 不允许在无数据时生成伪造的业务结论、访谈引语或统计结果。
- 不以每一种问卷维护一套完全独立的硬编码报告页面。
- 不在当前阶段建设付费模板市场、短信投放或复杂营销自动化。

## 4. 用户角色与权限

| 角色 | 核心任务 | 权限要求 |
| --- | --- | --- |
| 问卷创建者 | 创建、编辑、配置、发布、分析和导出问卷 | 可管理本人私有问卷；可管理其有权限的团队问卷。 |
| 团队协作者 | 共同维护团队问卷和查看结果 | 必须是对应团队成员；权限不能仅靠前端隐藏。 |
| 答题者 | 通过分享链接填写并提交答卷 | 根据问卷规则匿名或实名；只能提交允许的次数。 |
| 报告阅读者 | 阅读和下载最终分析报告 | 只能访问被授权的问卷报告或导出产物。 |
| 系统管理员 | 管理系统配置、模型和审计 | 当前不作为第一阶段用户界面范围，但需预留服务端治理能力。 |

### 4.1 权限原则

- `private` 问卷仅创建者可管理。
- `team` 问卷仅对应团队成员可访问。
- 获取答卷、生成 AI 报告和导出报告必须在服务端校验权限。
- 公开答题接口只允许读取已发布且处于有效时间窗口的问卷。
- 任何页面隐藏都不能替代接口权限校验。

## 5. 核心用户旅程

### 5.1 创建者主流程

1. 用户进入问卷工作台。
2. 用户从空白、内置模板、已保存模板或 AI 创建中选择入口。
3. 用户设置问卷标题、描述、作用域和题目。
4. AI 可根据目标或参考资料生成草稿，用户确认后才能应用。
5. 用户为题目分配业务分类，并配置报告章节和输出模块。
6. 用户设置匿名/实名、时间窗口、答卷上限、一人一答和提交文案。
7. 用户发布问卷并复制分享链接。
8. 答题者提交真实答卷。
9. 创建者查看概览、单份答卷和报告规划。
10. 系统按报告规划和分类生成报告，逐步展示生成进度。
11. 创建者复核结论并导出 PDF、Word 或图片。

### 5.2 零答卷流程

- 系统可以展示报告规划、章节结构、所需指标和待生成模块。
- 系统不得显示虚构样本、虚构百分比或确定性业务结论。
- 页面必须明确标注“等待答卷”或“仅展示分析框架”。
- 导出操作默认不可生成正式结论版报告；若允许导出，应带显著的草稿水印和数据限制说明。

## 7. 详细功能需求与验收标准

### 7.1 工作台与导航

#### SYS-WORKSPACE-001 问卷列表

需求：用户进入 `/surveys` 后，应看到其可访问的私有和团队问卷，并能识别标题、状态、答卷数、更新时间和作用域。

实现要求：

- 列表数据由服务端根据当前用户和团队权限返回。
- 默认按更新时间倒序。
- 支持“我的问卷”“团队问卷”“问卷模板”“AI 创建”切换。
- 空列表必须提供明确的创建入口。

验收标准：

- Given 用户拥有一份私有问卷，When 打开工作台，Then 列表显示该问卷且标注私有作用域。
- Given 用户不是某团队成员，When 获取列表，Then 不返回该团队问卷。
- Given 当前标签没有数据，Then 显示对应空状态，不能显示其他标签的数据。
- 页面刷新后保持由 URL 表达的模板视图状态。

当前状态：已实现。

#### SYS-WORKSPACE-002 五步工作流

需求：问卷编辑过程统一为“设计问卷、报告规划、发布回收、查看答题、分析报告”五个步骤。

验收标准：

- 当前步骤使用明确的高对比度选中态。
- 点击任一步骤可进入对应页面，且不会丢失当前问卷 ID。
- 页面刷新后根据 URL 恢复当前步骤。
- 小屏幕下步骤可以换行或横向滚动，不能被裁切或重叠。

当前状态：部分实现，流程存在，但路由表达和多套页面壳仍需统一。

### 7.2 模板系统

#### SYS-TEMPLATE-001 内置模板

需求：系统提供覆盖用户信息、满意度、市场调研、商品安全、活动反馈、员工反馈、教育反馈、NPS 和专业诊断场景的内置模板。

每个模板至少包含：

- 唯一 ID、名称、说明和业务分类。
- 预计完成时间和标签。
- 有顺序的题目集合。
- 题目分类建议。
- 默认报告规划。

验收标准：

- 模板库接口返回稳定的结构化数据。
- 应用模板后，标题、描述、题型、选项、必填和顺序与模板一致。
- 模板题目分类在保存问卷后仍可恢复，而不是只存在于当前浏览器内存。
- 内置模板不可被普通用户直接覆盖。

当前状态：部分实现，模板和前端分类可用，但题目分类持久化不完整。

#### SYS-TEMPLATE-002 用户模板

需求：用户可将当前问卷保存为可复用模板，并可再次应用或删除。

验收标准：

- 空标题或零题目模板返回 400 和字段级错误。
- 保存成功返回 201。
- 只有模板所有者或被授权团队成员可删除模板。
- 应用模板不得覆盖当前问卷，除非用户确认。

当前状态：部分实现；基础保存和读取存在，但当前分支缺少 `/api/survey-templates/:id` 删除路由，删除闭环、权限和版本治理需要补齐。

### 7.3 AI 创建与编辑

#### SYS-AI-001 从目标或资料生成问卷

需求：用户输入研究目标、受众、决策问题、约束或粘贴参考资料后，AI 返回结构化问卷草稿和报告规划。

AI 输出必须包含：

- `intentCanvas`：目的、受众、要支持的决策、所需信息和限制。
- `surveyDraft`：标题、描述、题目、题型、选项、必填和分类。
- `reportTemplate`：标题、章节、指标、图表槽位和限制。
- `clarifyingQuestions`：仅在缺失信息会显著影响质量时返回。

验收标准：

- 空指令返回 400，不发起模型调用。
- AI 返回内容必须经过 schema 校验；非法题型、空题目和重复选项必须被拒绝或清洗。
- 用户可先预览，再应用、保存或发布。
- AI 不得未经确认修改正式问卷。
- 模型失败时返回明确错误，或使用被标注的确定性回退草稿。
- 自动化测试不得依赖真实外部模型。

当前状态：部分实现。

#### SYS-AI-002 AI 变更集

需求：AI 对已有问卷的修改必须表现为可审阅的操作集合。

支持操作：新增、改写、删除、移动题目，修改选项、必填、题型、分类和报告规划。

验收标准：

- 每个操作有唯一 ID、操作类型、目标对象和变更前后值。
- 用户可逐条选择是否应用。
- 未确认的操作不能写入数据库。
- 部分操作失败时，系统必须返回失败项，不能静默丢失。

当前状态：部分实现，已有变更集界面和确认机制，分类持久化仍受数据模型限制。

### 7.4 问卷编辑器

#### SYS-EDITOR-001 基础编辑

需求：用户可编辑问卷元数据和题目结构。

验收标准：

- 标题为空时禁止保存并显示字段错误。
- 问卷至少包含一道有效题目。
- 需要选项的题型至少包含两个非空且不重复的选项。
- 上移、下移和删除后顺序立即更新，保存并刷新后仍保持一致。
- 切换预览不会丢失未保存内容，并明确提示未保存状态。

当前状态：部分实现；创建态编辑和数据访问层存在，但当前分支缺少单问卷读取、更新和删除路由。

#### SYS-EDITOR-002 题型能力

目标题型：短文本、长文本、邮箱、数字、电话、单选、多选、下拉、评分、线性量表、NPS、日期、时间和文件。

验收标准：

- 前端输入控件与题型语义一致。
- 服务端对必填、数据类型和选项合法性进行二次校验。
- 邮箱、数字、电话、日期和时间题不能只依赖浏览器校验。
- 文件题必须上传到受控存储，只在答卷中保存文件引用和元数据。

当前状态：部分实现；文件存储和部分服务端类型校验待补。

### 7.5 报告规划

#### SYS-PLAN-001 问卷绑定报告规划

需求：每份问卷创建时自动绑定一份默认报告规划，用户可在收集答卷前编辑规划。

报告规划至少包含：

- 报告标题和目标读者。
- 决策问题与成功标准。
- 有顺序的分类/章节。
- 每个章节关联的问题 ID。
- 每个章节的输出模块：图片、报表、文本。
- 指标定义、图表类型、图片提示要求和文本分析要求。
- 数据限制和低样本规则。

验收标准：

- 创建问卷后读取详情即可获得报告规划。
- 保存规划后刷新页面结构、顺序和模块保持不变。
- 删除题目时，引用该题目的章节必须提示修复，不能留下无效引用。
- 至少一个章节且每个章节至少一个输出模块。
- 最终报告严格按规划的章节、顺序和模块渲染。

当前状态：部分实现；基础模板已持久化，模块级规划模型待完善。

#### SYS-PLAN-002 AI 自动分类

需求：AI 根据题目语义和研究目标建议分类，但用户拥有最终决定权。

验收标准：

- 同一问题只能有一个主分类，可有多个辅助标签。
- AI 返回分类理由和置信度。
- 用户修改后的分类优先级高于再次生成的默认建议。
- 未被任何章节引用的问题必须在规划校验中显示。

当前状态：部分实现。

### 7.6 发布与答题

#### SYS-PUBLISH-001 发布规则

需求：创建者可配置发布状态、答题身份、起止时间、答卷上限、一人一答和提交确认文案。

验收标准：

- 未发布问卷拒绝提交并返回 409。
- 当前时间不在发布窗口内时拒绝提交。
- 达到答卷上限后拒绝新提交。
- 实名且一人一答时，同一用户第二次提交返回 409 和可理解错误。
- 匿名模式不能假装实现可靠的一人一答；若使用设备策略，必须说明其限制。
- 设置保存后列表、编辑器和答题接口读取一致。

当前状态：部分实现；提交拦截规则存在，但发布设置的更新接口在当前分支不完整。

#### SYS-ANSWER-001 答题与提交

需求：答题者通过分享链接填写问卷，系统校验后原子化保存一份答卷。

验收标准：

- 必填题未完成返回 400，并返回缺失题目 ID。
- 提交成功返回 201、答卷 ID 和提交时间。
- 重复点击提交不能产生重复答卷。
- 答题成功页显示问卷配置的确认文案。
- 移动端 360px 宽度下不存在横向溢出，控件可触达。

当前状态：已实现基础提交，幂等和移动端专项验证待补。

### 7.7 答卷和统计

#### SYS-RESULT-001 概览

需求：结果页展示问卷状态、答卷数、问题数、基础分布和数据质量提示。

验收标准：

- 数值与 `/responses` 接口返回一致。
- 单选、多选、评分、量表和 NPS 使用适合的统计方式。
- 文本题不绘制无意义的数值图表。
- 零答卷显示空状态，图表不使用模拟数据。

当前状态：已实现基础能力。

#### SYS-RESULT-002 单份答卷

需求：创建者可查看每份答卷的提交时间和题目答案。

验收标准：

- 只有问卷管理者可获取答卷列表。
- 答案按问卷题目顺序展示。
- 已删除或历史题目不能导致整个答卷页面崩溃。
- 敏感字段在导出和日志中遵循最小暴露原则。

当前状态：已实现基础能力。

### 7.8 AI 专业报告

#### SYS-REPORT-001 按分类生成

需求：后端读取报告规划中的分类，汇总各分类关联问题及其真实答卷，逐分类生成报告内容，并按规划顺序合并。

每个分类输入至少包含：

- 分类目标和管理决策问题。
- 关联问题、题型、选项和答案分布。
- 有效样本数、缺失率和必要的交叉信息。
- 该分类的输出模块和生成要求。
- 全局研究背景及其他分类的必要上下文。

验收标准：

- 分类数量和顺序与保存的报告规划一致。
- 每个分类只引用其关联问题，跨分类引用必须有明确理由。
- 生成过程按分类返回进度事件，已完成分类可立即预览。
- 某一分类失败不应丢失其他已完成分类；用户可单独重试。
- 页面刷新后恢复最近一次成功报告和分类状态。

当前状态：待实现完整的分类流式与持久化闭环。

#### SYS-REPORT-002 专业文本分析

需求：文本报告采用结论先行、证据支撑和行动导向的咨询式表达，但不得模仿或声称由特定咨询公司制作。

每个分析模块至少回答：

1. What：数据发生了什么。
2. So what：对当前决策为什么重要。
3. Now what：下一步应采取什么具体行动。

内容规则：

- 先写结论，再写证据、限制和行动。
- 引用具体题目、人数、比例、均值或主题频次。
- 区分已确认事实、数据推断、待验证假设和管理建议。
- 不同分类采用不同分析框架，例如人员画像、流程断点、数据治理、风险优先级、转化漏斗或满意度驱动因素。
- 建议必须包含对象、动作、时间范围和衡量指标。
- 不允许把相关性写成因果关系。

验收标准：

- 任意结论都能追踪到问题 ID 和计算数据。
- 零答卷时不出现“最高占比”“主要人群”等数据结论。
- 两种不同业务分类生成的分析结构和行动建议明显不同。
- 对低样本、缺失和偏差给出具体限制，不使用空泛免责声明。

当前状态：部分实现，仍需以分类专用提示词和证据引用 schema 强化。

#### SYS-REPORT-003 执行摘要与行动计划

需求：报告开头提供管理层可直接阅读的摘要，末尾提供可执行行动计划。

执行摘要应包含：核心结论、关键证据、主要风险、决策含义和优先行动。行动计划应包含负责人角色、时间、依赖、KPI 和复核节点。

验收标准：

- 只阅读摘要即可理解报告的主结论和下一步。
- 摘要中的数字与正文一致。
- 行动项不可使用“加强、提升、优化”作为唯一动作描述。
- 没有足够证据时，行动应表述为补充研究或验证，而非业务定论。

当前状态：部分实现，页面已有摘要展示，但内容质量依赖完整报告生成链路。

### 7.9 图片与图表

#### SYS-VISUAL-001 数据图表

需求：图表类型由题型、数据形态和报告规划共同决定。

验收标准：

- 类别分布可使用条形图、柱状图或环形图。
- 时间或阶段变化才使用折线图。
- 漏斗图只用于有明确顺序和可比口径的阶段数据。
- 每张图包含结论型标题、单位、样本口径和必要注释。
- 图表值与原始答卷聚合结果一致。

当前状态：部分实现，已有基础图表，但自动选型和专业注释不完整。

#### SYS-VISUAL-002 AI 配图

需求：当报告规划包含图片模块时，后端调用配置的图片模型生成与该分类业务语境相关的配图，并保存稳定 URL 或内部对象引用。

验收标准：

- 图片请求包含报告主题、分类、受众、视觉要求和禁止项。
- 图片生成成功后写入报告 artifact，刷新后仍可展示。
- 图片生成失败时显示可重试状态，不得用假图冒充结果。
- 导出文件包含与网页预览相同的图片。
- 图片来源、模型和生成时间可审计。

当前状态：待实现生产闭环；Wan 2.7 可作为目标提供方，但必须通过可替换的 provider 接口接入。

### 7.10 报告持久化与版本

#### SYS-ARTIFACT-001 报告产物

需求：每次正式生成都创建报告产物，保存报告结构、内容、数据快照、模型信息和状态。

目标最小字段：

- `id`, `survey_id`, `report_plan_version`, `status`。
- `content_json`, `source_snapshot_json`, `model_trace_json`。
- `generated_by`, `created_at`, `completed_at`, `error_message`。
- 分类级状态和重试次数。

验收标准：

- 刷新页面后读取最近一次成功产物。
- 新一轮生成不覆盖旧版本。
- 用户可以识别当前报告基于哪一版问卷、规划和答卷快照。
- 生成失败有持久化失败状态和可诊断错误。

当前状态：待实现完整数据闭环。

### 7.11 导出

#### SYS-EXPORT-001 CSV

需求：导出答卷原始数据，每行一份答卷，每列一个稳定问题标识。

验收标准：

- 使用 UTF-8 BOM 或其他兼容方式保证中文在 Excel 中正确显示。
- 逗号、引号和换行正确转义。
- 导出列包含答卷 ID、提交时间和问题 ID/标题映射。

当前状态：已实现基础下载，需补格式专项测试。

#### SYS-EXPORT-002 PDF

需求：导出与网页报告内容、顺序、颜色、图片和图表一致的正式 PDF。

验收标准：

- 文件是真实 PDF 格式，可被标准阅读器打开，不是带 PDF MIME 的纯文本。
- 中文字体嵌入或有可靠回退，不出现乱码。
- 所有内容在页面可打印区域内，不被横向裁切。
- 图片保持宽高比，最大宽度不超过内容区。
- 章节分页合理，标题不单独留在页尾。
- 与同一报告 artifact 的网页内容逐模块一致。

当前状态：待实现；现有文本下载不能通过此验收。

#### SYS-EXPORT-003 Word

需求：生成可编辑的 `.docx`，保持报告章节、图表图片、文本、颜色和基本布局。

验收标准：

- 文件通过 Office 和 WPS 打开无修复提示。
- 页面尺寸、页边距和内容宽度固定，图片不超出正文区域。
- 图表可作为高分辨率图片嵌入，文字保持可编辑。
- 内容来自当前报告 artifact，不重新拼接另一套报告文案。

当前状态：待实现正式闭环。

#### SYS-EXPORT-004 图片

需求：将完整报告或单个章节导出为清晰图片。

验收标准：

- 导出宽度有上限，长报告按章节拆分或生成长图。
- 背景色、图表颜色和图片与网页一致。
- 2x 像素密度下文字仍清晰。
- 不包含应用导航、按钮、悬浮助手等非报告内容。

当前状态：待实现正式闭环。

## 9. 数据模型目标

现有核心实体：`surveys`、`survey_questions`、`survey_responses`、`survey_templates`、`survey_report_templates`。

需要补齐或明确的实体：

| 实体 | 目的 | 关键关系 |
| --- | --- | --- |
| `survey_question_metadata` 或题目 `metadata` | 保存题目分类、标签和分析角色 | 每题一份或 JSONB 字段 |
| `survey_report_plans` | 保存模块级报告规划和版本 | 每份问卷多版本 |
| `survey_ai_sessions` | 保存 AI 创建、优化和报告生成会话 | 关联用户、问卷和模型 |
| `survey_ai_model_traces` | 保存提示词版本、模型、延迟和错误，不保存不必要敏感数据 | 关联 session |
| `survey_report_artifacts` | 保存最终结构化报告和数据快照 | 关联问卷和规划版本 |
| `survey_report_assets` | 保存 AI 图片、图表快照和导出资源 | 关联 artifact 和 block |

数据一致性要求：

- 问卷、题目和新建默认规划应在同一事务或可补偿事务中完成。
- 报告产物引用固定的数据快照，不能随答卷新增而静默改变历史报告。
- 删除问卷时定义答卷、规划、产物和资源的级联或软删除策略。
- AI 原始响应不得直接作为前端唯一数据源，必须清洗为稳定 schema。

## 10. AI 提示词与智能分析要求

### 10.1 系统级约束

- AI 的身份是研究与决策分析助手，不是自由写作助手。
- 报告规划决定章节、顺序、模块和数据来源；AI 只填充内容，不擅自改结构。
- 只使用请求中提供的问卷、答卷和背景，不引入未经标注的外部事实。
- 输出必须符合 JSON schema，并包含证据引用。
- 数据不足时输出待验证假设和补充研究建议。

### 10.2 分类级分析策略

| 分类 | 重点问题 | 建议框架 |
| --- | --- | --- |
| 人员/画像 | 样本由谁构成，关键群体是否被覆盖 | 结构、代表性、分层差异、覆盖缺口 |
| 流程 | 哪些环节低效、重复或适合自动化 | 流程链、断点、时长、标准化程度、控制点 |
| 数据 | 数据是否可获取、可信、可治理 | 完整性、准确性、及时性、权限、主数据与血缘 |
| 问题/风险 | 哪些痛点最重要、最紧急 | 频率、影响、可控性、风险等级和优先级矩阵 |
| 落地/牵引 | 组织是否具备规模化实施条件 | 场景价值、可行性、采纳、治理、资源与里程碑 |
| 满意度/NPS | 满意与推荐由什么驱动 | 分数、分布、驱动因素、流失风险和闭环动作 |
| 购买/转化 | 用户处于什么阶段、阻力在哪里 | 认知、考虑、试用、购买、复购和流失漏斗 |
| 开放文本 | 用户反复表达哪些主题 | 主题聚类、频次、情绪、代表性证据和未解决问题 |

### 10.3 输出质量门槛

- 每个核心结论包含 `claim`、`evidence`、`implication`、`confidence` 和 `sourceQuestionIds`。
- 建议包含 `action`、`ownerRole`、`timeframe`、`metric` 和 `dependency`。
- 风险包含发生条件、影响和缓解措施。
- 不同分类不能套用同一组句式，仅替换题目名称和数字。

## 11. 非功能需求

### 11.1 可用性与响应式

- 关键文字与背景对比度至少满足 WCAG AA。
- 选中、悬停、禁用、加载、成功和失败状态必须可识别。
- 1280px、1440px、1920px 和 390px 视口下无主要内容遮挡或横向溢出。
- 数字、长标题和中文长词不能撑破按钮、卡片和侧栏。

### 11.2 性能

- 普通问卷列表和编辑页面的主要内容在合理网络条件下 2 秒内可交互。
- 100 题和 1000 份答卷时，结果页不应一次性渲染全部单份答卷。
- AI 报告使用异步任务和分类级增量展示，避免单请求长时间无反馈。

### 11.3 安全与隐私

- 所有管理接口执行服务端权限校验。
- 模型调用前对个人信息执行最小化和必要脱敏。
- 密钥只存在于服务端环境变量。
- 导出文件和报告资源使用授权下载或有过期时间的签名 URL。
- 对上传文件校验类型、大小并执行恶意内容检测。

### 11.4 可观测性

- AI 请求记录模型、提示词版本、耗时、token/费用信息和结果状态。
- 报告生成有任务 ID、分类进度、失败原因和重试记录。
- 关键接口有结构化日志和错误率指标。

## 12. 异常与降级规则

| 场景 | 预期行为 |
| --- | --- |
| 无答卷 | 只展示规划和空状态，不生成事实性结论。 |
| 低样本 | 允许方向性分析，但显示样本数、覆盖限制和置信度。 |
| AI 超时 | 保存失败状态，保留已完成分类，允许重试。 |
| AI 返回非法 JSON | 进行 schema 清洗；无法修复则失败并记录原始错误摘要。 |
| 图片生成失败 | 文本和图表继续生成，图片模块显示失败与重试。 |
| 导出失败 | 不产生损坏文件，显示错误并保留报告预览。 |
| 题目被删除 | 报告规划标记失效引用，要求用户修复后再生成。 |
| 发布期结束 | 保留历史答卷和报告，阻止新提交。 |
| 重复生成 | 创建新版本，不覆盖最近一次成功报告。 |

## 13. 端到端验收场景

### AC-E2E-001 模板创建完整闭环

1. 用户登录并进入模板库。
2. 选择专业模板并创建问卷。
3. 编辑题目并保存。
4. 验证题目分类和默认报告规划刷新后仍存在。
5. 发布问卷并提交至少三份答卷。
6. 生成报告并验证每个分类只使用关联问题。
7. 刷新页面并恢复同一报告。
8. 导出 PDF 和 Word，内容与网页模块一致。

通过条件：以上步骤全部成功，且数据库中存在问卷、题目、分类、答卷、规划版本和报告产物。

### AC-E2E-002 AI 创建完整闭环

1. 用户输入目标和参考资料。
2. AI 返回 intent canvas、问卷草稿和报告规划。
3. 用户拒绝一个变更、接受其余变更。
4. 保存后正式问卷只包含已接受内容。
5. 发布并收集答卷。
6. 报告按保存规划生成，而非按 AI 临时改变结构。

通过条件：AI 不产生隐藏写入，所有正式变化可追踪到用户确认。

### AC-E2E-003 零答卷安全性

1. 创建并发布问卷，不提交答卷。
2. 打开报告页并点击生成。
3. 检查页面和接口结果。

通过条件：响应包含报告规划和限制；不包含虚构人数、比例、引语、主要群体或业务结论。

### AC-E2E-004 权限隔离

1. 用户 A 创建私有问卷和团队问卷。
2. 非团队成员用户 B 尝试读取、修改、获取答卷、生成和导出报告。

通过条件：所有管理接口返回 403 或 404，且没有敏感数据泄漏。

### AC-E2E-005 导出一致性

1. 生成包含图片、报表和文本的报告。
2. 分别导出 PDF、Word 和图片。
3. 按模块对比网页和导出结果。

通过条件：模块数量、顺序、标题、正文、颜色和视觉资产一致；图片不超宽，中文不乱码，文件可正常打开。

4. AI 图片生成、资源存储和恢复。
5. 真正的 PDF、Word 和图片导出。

## 16. AI Agent 使用规则

后续 AI 在分析、拆分或实现该系统时必须遵守：

1. 先读取本文档，再读取当前代码和测试；本文档中的“当前状态”可能随实现变化，需要用代码重新验证。
2. 不得把“部分实现”直接宣布为完成。
3. 每个开发任务必须引用一个需求编号和至少一条验收标准。
4. 不得为通过演示而使用模拟答卷冒充真实数据。
5. 修改报告生成时必须验证报告规划结构保持不变。
6. 修改导出时必须使用当前报告 artifact，不维护另一份独立文案生成逻辑。
7. 完成任务时必须报告验证命令、结果和仍存在的风险。

## 17. 完成定义

一项需求只有同时满足以下条件才可从“部分实现/待实现”改为“已实现”：

- 用户可完成完整主流程。
- 服务端数据和权限闭环存在。
- 正常、空状态、失败和重试行为明确。
- 页面刷新后关键状态不丢失。
- 至少一项自动化测试覆盖核心行为。
- 验收标准有可重复证据。
- 未以 Mock、占位文案、模拟图表或 MIME 伪装替代正式能力。
