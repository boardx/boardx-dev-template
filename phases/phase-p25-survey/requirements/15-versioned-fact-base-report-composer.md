# 15 - 版本化事实库与精简报告编排器

## 事实来源

- 用户于 2026-07-18 对 `/surveys?survey=<id>&step=template` 实际页面提出变更：
  中间栏把设置与预览纵向堆叠后过长；指定问题和图片/图表/文本三套设置过于复杂。
- 标注截图：
  `ui-preview/2026-07-18-report-composer-length-review.png`。
- 用户确认报告配置只保留自然语言约束，由系统基于整份问卷和全部答卷自主选择所需证据。
- 用户确认报告不应持续自动生成，采用“版本化事实库快照 + 按需生成 + 不可变报告版本”。
- 专业报告方法参考公开的咨询研究实践：建立一致事实基础、记录并验证假设、形成执行摘要；
  不声称复刻任何机构未公开的内部 AI 系统。
- Umbrella issue：`boardx/boardx-dev-template#648`。本 feature 必须使用独立 feature issue
  和独立 Delivery PR，PR 中以 `Refs #648` 关联总需求。

## 当前问题

- 报告章节配置要求用户逐题选择来源，并分别配置图片、图表和文本模块，重复表达同一个报告意图。
- 设置和图表预览在同一窄栏纵向排列，页面过长，用户无法同时比较约束与结果。
- 当前报告配置绑定 `questionIds`、`inputModes`、图表类型和映射 JSON，阻碍系统从整份答卷中自主寻找证据。
- 报告生成缺少稳定的数据修订号和产物复用键，无法可靠区分“数据已更新”与“必须重新生成”。

## 已确认用户体验

### 1. 双栏报告工作区

- 左侧保留报告章节列表和排序。
- 中间工作区改为横向双栏：左侧“报告要求”，右侧“报告预览”；桌面端同屏比较。
- 窄屏按顺序折叠为设置、预览，不产生横向溢出。
- 现有常驻 AI 推荐助手不再占据第三列，改为可折叠的生成记录与证据面板。

### 2. 精简报告要求

- 删除逐题绑定、可添加问题、自由模块堆叠和可编辑映射 JSON。
- 每章必须且只能选择一种输出：图片、图表或文本。
- 图表使用白名单 ECharts 官方模板，右栏提供效果预览和只读 Option JSON。
- 自然语言要求描述分析约束；系统仍从整份问卷事实库检索证据。

### 3. 按需生成与版本状态

- 首次没有报告时显示“生成报告”。
- 问卷或答卷未变化且要求未变化时，打开页面直接复用现有报告，不调用模型。
- 新答卷或问卷结构变化时，现有报告仍可阅读，但显示“数据有更新”和新增样本数，不自动生成。
- 用户点击“更新报告”后才生成新版本；旧版本不可变并可从版本记录中查看。
- 同一事实库修订、同一报告要求和同一模板版本重复请求时，返回已有成功产物。

## 版本化事实库契约

### Source revision

- 服务端把一份问卷的元数据、完整题目定义、发布口径和全部授权答卷转换为只读事实库快照。
- 快照逻辑路径为 `survey-source.jsonl`，但不要求落到 Web 服务器宿主机文件系统。
- 每个快照至少包含：
  - `surveyId`
  - `sourceRevision`
  - `contentHash`
  - `schemaVersion`
  - `generatedAt`
  - `sample.validCount`
  - 问卷、题目、匿名化答卷及统计口径记录
- 只有问卷内容、影响分析的发布口径或答卷集合变化时才创建新 `sourceRevision`。
- 重复打开页面、重复预览或重复点击生成不会为相同内容创建新快照。

### Report artifact

- 报告产物使用稳定键：
  `sourceRevision + requirementHash + templateVersion`。
- 成功产物不可原地覆盖；更新生成新的 `reportVersion`。
- 产物记录来源修订、样本量、生成时间、约束摘要、状态、证据索引和失败原因。
- 生成失败不得覆盖最近成功版本；用户可以重试失败任务。

### Staleness

- `currentSourceRevision === artifact.sourceRevision` 时报告为最新。
- 修订不同且已有成功产物时报告为“数据有更新”，继续展示旧报告并提示差异。
- 约束变化但未生成时显示“要求已修改”，不把旧产物误标为新要求的结果。

## 数据与安全约束

- 原始完整答卷只在服务端授权边界内进入事实库，不发送到浏览器。
- 浏览器只接收生成状态、聚合指标、允许展示的匿名证据和报告文档。
- 继续执行现有 `canView` / `canManage`、owner/team/room 权限顺序。
- 开放题引用必须匿名化；不得把身份字段混入生成提示或报告证据。
- 零样本不得生成虚构数字或业务结论；低样本必须展示限制。

## 兼容与迁移

- 读取既有分类报告配置时，把旧的模块提示合并为自然语言报告要求；旧字段在迁移期只读兼容。
- 读取旧章节时，缺失或无效的 `outputType` 必须归一化为 `text`；迁移期 `inputModes` 只保留
  一个与归一化输出类型对应的单值兼容投影，不得继续表达多输出；新保存不依赖该字段。
- `requirementHash` 必须由归一化后的章节标题、唯一输出类型（`image`、`chart` 或 `text`）、
  可选的白名单 `chartTemplateId` 和归一化后的自然语言要求共同计算；未选择图表模板时省略
  `chartTemplateId`。
- 持久化只接受产品维护的白名单 `chartTemplateId`。任意 ECharts `option`、脚本、
  `formatter` 函数和外部 URL 都必须拒绝，且不能通过浏览器提交绕过服务端校验。
- 新保存不再要求 `questionIds`、`inputModes`、图表样式或映射字段。
- 既有成功报告继续可查看，不因新契约迁移而丢失。

## F16 范围

- 实现双栏工作区和精简报告要求。
- 实现事实库修订、内容哈希、报告产物复用、过期提示和不可变版本记录。
- 使用现有服务端证据聚合/确定性生成路径完成可验收闭环。
- 定义供 F17 使用的事实库读取接口，但本 feature 不引入 Deep Agents 或 LangGraph 依赖。

## 非范围

- Deep Agents/LangGraph 文件系统和多分析模块编排属于 F17。
- Word、真实 PDF、PNG 等专业导出仍属于 F13。
- 不改变公开答题协议、问卷生命周期或发布权限。

## 验收

- 报告编辑页不再出现逐题选择、自由模块堆叠或可编辑 JSON 映射；每章只选择图片、图表或文本之一。
- 图表使用白名单 ECharts 官方模板，右栏提供效果预览、完整只读 Option JSON 和复制操作；自然语言要求仍由整份问卷事实库提供证据。
- 官方样例值只能用于草稿预览，绝不能进入事实库、证据或报告产物。
- 桌面端设置与预览同屏，生成记录/证据面板可折叠；移动端无横向溢出。
- 相同 `sourceRevision + requirementHash + templateVersion` 不创建重复任务或产物。
- 新答卷只标记报告过期；模型调用次数保持不变，直到用户点击“更新报告”。
- 更新成功后产生新版本，旧版本仍可访问；失败时保留最近成功版本。
- 浏览器响应不包含完整原始答卷。
