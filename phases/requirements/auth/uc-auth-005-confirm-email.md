Use Case 名称：
确认邮箱

Actor：
访客、注册用户

目标：
用户通过邮箱确认入口完成账号邮箱验证或新邮箱确认。

系统边界：
BoardX / 账号与身份

前端入口：
1. 邮件中的确认链接。
2. 邮箱确认页面。
3. 新邮箱确认页面。

前置条件：
- 用户收到邮箱确认链接。
- 确认链接中包含系统可识别的确认信息。

触发条件：
用户点击邮件中的确认链接，或直接打开确认邮箱页面。

主流程：
1. 用户点击邮箱中的确认链接。
2. 系统打开 confirm-email 或 confirm-new-email 页面，并读取链接中的 hash 参数。
3. confirm-email 页面展示 BoardX 标识、加载动画和 verifying 文案；confirm-new-email 页面展示加载动画。
4. 系统提交 hash 进行确认，用户在处理期间只看到加载状态。
5. 注册邮箱确认成功后，系统展示邮箱确认成功提示，并跳转到 profile 页面。
6. 新邮箱确认成功后，系统展示邮箱确认成功提示；如果当前用户已登录，系统刷新当前用户资料并跳转到 profile 页面，否则跳转到公开入口。
7. 确认失败时，系统展示确认失败提示，并跳转到公开入口。

备选流程：
- A1：用户已登录并确认新邮箱，系统更新账号邮箱后返回个人资料页。
- A2：用户未登录并确认注册邮箱，系统展示成功状态后引导登录。
- A3：用户重复打开已使用过的确认链接，系统展示已确认或链接不可重复使用的状态。

异常流程：
- E1：确认链接缺少必要信息，系统展示链接无效。
- E2：确认信息过期，系统提示重新发起确认。
- E3：确认信息与当前用户或目标邮箱不匹配，系统提示确认失败。
- E4：服务异常，系统展示确认失败并允许用户稍后重试。

权限与可见性：
1. 访客可以打开确认邮箱页面，但只能完成链接中指定账号的确认动作。
2. 已登录用户确认新邮箱时，系统必须确保该链接属于当前用户或允许的邮箱变更流程。
3. 确认页面不应展示其它账号的敏感信息。

后置条件：
- 成功时，邮箱确认状态或新邮箱绑定状态被更新。
- 失败时，账号邮箱状态保持不变。

不包含：
1. 不包含 Team、Room 或 Board 业务能力。
2. 不包含邮件投递流程。

业务规则：
1. 邮箱确认入口应有有效期。
2. 邮箱变更确认与注册确认应区分处理。
3. 无效或过期链接不能更新邮箱状态。
