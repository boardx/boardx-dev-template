Use Case 名称：
AI 引用知识库上下文

Actor：
普通用户、Team 成员、Agent/AI Tool 创建者、AI 服务

目标：
用户让 AVA、Agent 或 AI Tool 在回答时使用当前上下文中已完成处理的知识库文件。

系统边界：
BoardX / 知识库

前端入口：
1. Personal Knowledge Base 页面 > completed 文件。
2. Team Knowledge Base 页面 > completed 文件。
3. Agent 创建器 > Knowledge 区域。
4. AI Tool 创建器 > Knowledge 区域。
5. 相关聊天或工具执行入口。

前置条件：
- 用户已登录。
- 知识库文件已上传并处于 completed 状态。
- AI 服务可读取当前上下文允许的知识文件。

触发条件：
用户在 AVA、Agent 或 AI Tool 场景中提问或执行工具，并使用相关知识库上下文。

主流程：
1. 用户在个人、团队、Agent 或 AI Tool 场景中打开知识库区域，系统只加载当前上下文的文件，并展示文件状态。
2. 系统展示每个文件的处理状态；completed 文件表示已完成处理，可作为 AI 上下文候选。
3. 用户上传新文件后，系统自动处理并刷新文件列表；processing 文件会保留在列表中并定时刷新状态。
4. 用户进入 AVA、Agent 或 AI Tool 提问或执行场景，系统依据当前个人、Team、Agent 或 AI Tool 上下文关联可用知识文件。
5. 用户发送问题或执行请求时，系统只把当前上下文中已完成处理且用户有权访问的知识文件纳入 AI 服务可用范围。
6. 如果命中相关内容，AI 服务可结合知识库内容生成回答；如果没有命中，AI 仍按普通聊天或工具上下文回答。
7. 如果某个文件仍在 processing 或 error 状态，系统不应把该文件描述为已可用知识上下文。
8. 用户删除文件后，后续 AI 请求不再使用该文件内容。

备选流程：
- A1：用户只使用个人知识库。
- A2：用户使用团队级知识库。
- A3：Agent 或 AI Tool 创建者使用专用知识文件。

异常流程：
- E1：文件尚未处理完成，系统跳过该文件或继续展示 processing 状态。
- E2：检索不到相关内容，AI 按普通上下文回答。
- E3：跨团队或跨资源上下文不匹配，系统禁止引用。

权限与可见性：
1. 普通用户只能让 AI 使用自己的个人知识库。
2. Team 成员只能让 AI 使用当前 Team 上下文中自己有权访问的知识文件。
3. Agent/AI Tool 创建者只能让对应资源使用其有权管理的知识文件。
4. 系统管理员不能通过普通聊天自动引用任意 Team 的私有知识文件。
5. AI 服务不能跨 Team、跨 Agent 或跨 AI Tool 主动读取未传入的知识文件。

后置条件：
- 成功时，AI 回答可使用当前上下文允许的知识库内容。

不包含：
1. 不包含文件存储实现。
2. 不包含知识检索的内部处理机制。
3. 不包含 AI 模型推理过程。

业务规则：
- 知识库上下文必须按 personal、team、agent、tool 等作用域隔离。
- 不得跨团队泄露文件内容。
- 文件处理状态影响是否可用于检索。
