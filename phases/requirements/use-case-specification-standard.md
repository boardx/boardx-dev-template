# BoardX Use Case 细化规范

本文档用于约束 BoardX Use Case 的细化粒度。当前 Use Cases 主要依据 `boardx-web` 的前端路由、页面组件、菜单配置、状态切片和用户可见交互来定义。

## 用例边界

每个 Use Case 必须回答：

- 谁在使用这个能力？
- 这个 Actor 要完成什么业务目标？
- 这个目标从哪个前端入口触发？
- Actor 与系统之间发生哪些交互？
- 哪些角色可以做，哪些角色只能看，哪些角色不能进入？
- 成功后系统中哪些业务状态发生变化？
- 哪些异常状态需要被 UI 明确表达？
- 从交互流程、权限和后置条件中能否自然推导界面行为、业务边界和端到端用户路径？
- 这个用例不包含什么？

Use Case 不写：

- 组件内部实现。
- 接口路径。
- 数据库字段。
- Redux action 名称。
- Yjs 或 HTTP 的内部同步细节。
- 具体 React 组件渲染逻辑。

但 Use Case 可以写：

- 前端入口位置，例如 Board Header、Board Menu、Widget Menu、Context Menu、Room 页面、Team 设置页。
- 可见状态，例如按钮隐藏、禁用、loading、空状态、权限不足、同步异常。
- 业务级状态变化，例如 Board 从私有变公开、用户从访客变 Board Visitor、组件从未锁定变锁定。

Use Case 不需要把验收内容拆成独立的技术验证章节。如果相关内容需要单独解释，通常说明主流程、备选流程、异常流程、权限与可见性、后置条件或业务规则还不够清楚。

Use Case 也不负责视觉设计。它不定义颜色、间距、排版、组件样式或页面美术方向，但必须准确表达界面交互事实：有多少界面或子界面、每个界面能看到什么、能执行什么操作、操作后出现什么状态或结果。

## 推荐粒度

### L0：系统级能力

用于画顶层图，例如：

- 账号与身份
- Team
- Room
- Board
- AVA
- AI Store
- 知识库
- 问卷
- 后台管理

L0 只用于导航和系统边界，不直接写成详细 Use Case。

### L1：模块级能力

用于模块访问权限图，例如：

- Board Header
- Board Menu
- Widget Menu
- Context Menu
- Canvas
- Room Chat
- Team Credits
- Survey Report

L1 描述角色能否进入该模块。

### L2：用户目标级 Use Case

这是主要 Use Case 文件的粒度，例如：

- 分享 Board
- 创建连接线
- 使用右键菜单调整层级
- 使用 Widget Menu 锁定组件
- 使用 Yjs 多人同步 Board 内容
- 加入公开 Room
- 发布问卷

L2 必须有完整模板。

### L3：流程变体或规则细节

只有当差异会影响界面行为、权限或业务结果时才拆为独立用例，例如：

- 私有 Board 访问
- 公开 Board 访问但未加入
- 公开 Board 访问并加入
- Board Visitor 只读导出
- Board Member 编辑画布

## 单个 Use Case 必填字段

```text
Use Case 名称：
Actor：
目标：
系统边界：
前端入口：
前置条件：
触发条件：
主流程：
备选流程：
异常流程：
权限与可见性：
后置条件：
不包含：
业务规则：
```

说明：

- `系统边界`：说明能力属于 BoardX 哪个模块，例如 Board / Canvas / Room。
- `前端入口`：说明用户从哪里触发，例如 Board Header > Share。
- `主流程`：必须从用户实际访问界面开始，写清楚用户看到什么、能点击什么、能输入/选择/上传什么，以及系统如何反馈状态和结果。
- `权限与可见性`：明确 owner/admin/member/visitor 的差异。
- `不包含`：防止需求蔓延。
- `业务规则`：写影响用户路径、权限判断、状态变化和异常结果的业务规则。

## 权限描述规范

权限必须按模块内角色写清楚：

### Team

- Team Owner：拥有 Team 最高管理能力。
- Team Admin：拥有 Team 管理能力，但不能替代 Owner 的最终所有权。
- Team Member：使用 Team 工作区能力，不默认拥有管理权限。

### Room

- Room Owner：拥有 Room 最高管理能力。
- Room Admin：管理 Room 成员和 Room 设置。
- Room Member：访问 Room、Room Chat、Room 文件和 Room 内 Board。
- 公开访问用户：能看到公开 Room，但需要主动加入后才成为 Room Member。

### Board

- Board Owner：拥有 Board 最高管理能力。
- Board Admin：管理 Board 权限、内容和设置。
- Board Member：编辑 Board 内容。
- Board Visitor：只读访问，不能修改内容。
- 公开访问用户：能看到公开 Board，但是否能加入取决于 Board 策略。

### Survey

- Survey Owner：创建者，拥有该问卷管理能力。
- Survey Admin：Team Owner/Admin 或被授权的问卷管理者。
- Survey Member：可创建或访问自己权限范围内的问卷。
- Respondent：只填写和提交问卷答复。

## 可推导性判断

一个合格的 Use Case 应该让界面行为、业务边界和端到端用户路径从用例内容中自然推导出来，而不是额外写成独立章节。

## AI Ready 判定标准

AI Ready 的 Use Case 不是更长的说明书，而是一个足够明确的用户交互规格。AI 读取单个 `uc-*.md` 后，应该能够直接推导：

- 页面或模块需要出现哪些入口、按钮、菜单、列表、卡片、弹窗、表单字段、状态提示和空状态。
- 用户能执行哪些动作，例如点击、输入、选择、拖拽、上传、下载、搜索、筛选、确认、取消、复制、删除、移动、缩放、锁定、分享、订阅或审批。
- 每个动作之后系统必须给出什么可见反馈，例如选中、高亮、加载、处理中、成功、失败、空状态、无权限、禁用、隐藏、同步中或完成提示。
- 不同 Actor 在同一入口下的差异：谁能进入、谁能查看、谁能编辑、谁只能看到禁用态、谁完全看不到入口。
- 成功后业务对象发生什么用户可感知变化，例如 Board 被创建、组件被锁定、连接线样式改变、Agent 被订阅、问卷被发布、文件进入知识库。
- 失败、取消、无权限、空数据、冲突、锁定、额度不足、网络失败时，页面应该保持什么状态并显示什么反馈。
- 从 `主流程`、`备选流程`、`异常流程`、`权限与可见性`、`后置条件` 能自然得到人工验证或自动化验证路径，而不需要额外猜测。

AI Ready 的最低质量门槛：

- `主流程` 不能只写“用户完成操作”；必须同时包含用户看到的界面元素、用户动作、系统反馈和结果状态。
- `前端入口` 必须定位到可感知入口，而不是泛泛写“进入系统”。
- `备选流程` 必须描述至少一个真实分支，例如取消、空状态、多选、只读、锁定、入口隐藏或数据为空。
- `异常流程` 必须描述失败后系统如何保留或恢复用户可见状态。
- `权限与可见性` 必须按该模块的角色差异写清楚，不能只写“按权限控制”。
- `业务规则` 必须约束会影响界面、状态或验证路径的规则。
- 不使用“待确认”“应该有”“可能有”“产品策略待定”等会阻断推理的模糊表达；未展示的入口一律不作为当前可操作能力。

## 界面-操作-结果可推导标准

`interface-operation-inventory.md` 是全系统界面索引，用于记录当前产品界面数量、界面入口、界面内主要可见内容、用户可执行操作和操作后的可见结果。单个 Use Case 与该清单配合后，必须能回答：

- 用户从哪个页面、面板、菜单、弹窗、列表、卡片或画布区域开始操作。
- 该界面当前能看到哪些主要内容、状态、入口、空状态、禁用态或只读态。
- 用户可以执行哪些明确动作，包括点击、输入、选择、搜索、筛选、拖拽、缩放、上传、下载、复制、删除、锁定、分享、订阅、审批、发布或取消。
- 每个动作之后用户能看到什么变化，例如列表刷新、卡片状态改变、弹窗打开或关闭、选中框出现、组件样式改变、消息追加、进度显示、成功提示、失败提示、权限不足、空状态或加载状态。
- 不同角色在同一界面下看到的入口和允许的动作有什么差异。

因此，Use Case 不需要写“UI 怎么设计”，但必须写清楚“用户实际看到什么、能做什么、做完后系统怎样表现”。如果只能从用例中知道模块名称，而不能知道界面内操作和操作结果，这个 Use Case 还没有达到 AI Ready。

界面行为可推导：

- 从 `前端入口`、`主流程`、`备选流程`、`异常流程` 和 `权限与可见性` 可以知道页面需要展示哪些入口、状态、反馈和限制。
- 例如 Board Visitor 打开 Board 时只能看到查看入口；Board Member 选中文本组件时可以看到文本样式入口；图片组件才出现裁剪能力。

业务边界可推导：

- 从 `Actor`、`前置条件`、`触发条件`、`业务规则` 和 `后置条件` 可以知道需要哪些业务对象、权限判断、成功结果和失败结果。
- 不在 Use Case 中写具体接口路径、数据库字段或 service 名称。

端到端用户路径可推导：

- 从 `主流程` 可以得到正常用户路径。
- 从 `备选流程` 可以得到分支用户路径。
- 从 `异常流程` 和 `权限与可见性` 可以得到无权限、失败、空状态和冲突路径。

不好的写法：

- 把“调用某 API”“写入某 collection”“更新某 React state”当成 Use Case 步骤。
- 只写“用户完成操作”，但不写入口、权限差异、成功状态和失败状态。

## 文件组织规范

大模块必须拆子目录：

```text
board/
  access/
  header/
  board-menu/
  widget-menu/
  context-menu/
  canvas/
  collaboration/
  widgets/
  local-workspace/
```

命名规范：

```text
uc-<module>-<序号>-<动词短语>.md
```

例如：

```text
uc-widget-menu-003-lock-unlock-widget.md
uc-context-menu-005-export-selected-content.md
uc-canvas-008-keyboard-operate-widget.md
```
